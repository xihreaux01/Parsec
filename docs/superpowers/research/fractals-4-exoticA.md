# Per-Fractal Guide Research: EXOTIC group A

Research + schema audit for four exotic fractals in Parsec Hyperdrive. Ground
truth is the live source (state schema + GLSL DE core), cited file:line. Web
research verifies the math and supplies community-known good parameters. Items
that could not be verified against a primary source are marked "unverified."

Scope note: per the task, only fractal-specific schema parameters are audited.
The shared Palette / Reflection / Light / Camera groups are intentionally
ignored. Cut-plane params (Cut, plane normal x/y/z, plane offset) and the shared
Quality params (Iterations, DE fudge, Bailout where it is purely a render knob)
are listed for completeness but treated as engine knobs rather than the
fractal's mathematical identity.

## Apollonian Gasket (`FractalType.Apollonian`)

**What it is:** The 3D Apollonian gasket (Coxeter tetrahedral sphere packing) is
the limit set of the Kleinian group generated by inversions through five
mutually tangent spheres: four unit spheres at the vertices of a regular
tetrahedron plus one bounding sphere. Repeated inversion packs ever-smaller
tangent spheres into the gaps, producing a fractal surface of Hausdorff
dimension approximately 2.474. A planar cross-section of the 3D packing is the
classic 2D Apollonian gasket (dimension approximately 1.3057).

**How Parsec computes it:** The DE core renders the limit set itself, not the
sphere surfaces. `estimate()` (apollonian_core.glsl:82) iterates inversions: for
each point `q`, if it lies inside any of the four inner spheres it is inverted
through that sphere with `k = r2_in / d2` and `logScale += log(k)`
(apollonian_core.glsl:112-124); otherwise if it is outside the bounding sphere
it is inverted back inside (apollonian_core.glsl:127-136); if neither fires the
orbit has settled in the free region and the loop breaks
(apollonian_core.glsl:138). The distance estimate is
`de = deEnvelope * exp(-logScale)` (apollonian_core.glsl:156): far from the limit
set (few inversions, logScale near 0) the DE is large so rays pass through the
outer shells; deep in the orbit (logScale large) the DE collapses to 0 so the ray
hits the actual gasket. Inner-sphere centres are the canonical regular-tetrahedron
vertices at distance sqrt(3/2) from the origin (the 1/sqrt(2) triples,
apollonian_core.glsl:63-67), each scaled by `tangency` at use
(apollonian_core.glsl:73-80). The bounding-sphere radius is the canonical Soddy
value `R_OUTER = (sqrt(6)+2)/2 = 2.2247...` (apollonian_core.glsl:71) scaled by
`outerRadiusMult`. Half-cut is a CSG intersection with a tilted half-space
(`de = max(de, plane)`, apollonian_core.glsl:164-168).

**Canonical formula / math background:** Sphere inversion through a sphere of
radius r centred at c maps q to c + (r^2 / |q-c|^2)(q-c). The four-unit-sphere
tetrahedral configuration and its bounding sphere follow from the 3D Descartes /
Soddy circle theorem generalized to spheres (curvatures of mutually tangent
spheres satisfy a quadratic relation; Soddy 1936). The limit set of inversions
in four tetrahedrally-arranged spheres is a Poincare-ball boundary fractal with
tetrahedral symmetry. Fractal dimension of the 3D Apollonian packing is
approximately 2.4739465 (researchgate / arXiv estimates), and the 2D gasket is
approximately 1.305688.

## Phoenix (`FractalType.Phoenix`)

**What it is:** The Phoenix set is a Julia-type fractal with memory: the next
iterate depends on both the current and the previous iterate, giving organic,
curling "flame/feather" growth instead of crystalline Julia structure. It was
introduced by Shigehiro Ushiki (IEEE Transactions on Circuits and Systems). The
canonical 2D map is z_{n+1} = z_n^2 + c + p*z_{n-1}, with the memory coefficient
p controlling how strongly the previous term feeds back. Parsec lifts this to 3D
using Mandelbulb-style n=2 trig.

**How Parsec computes it:** Iteration is
`z_{n+1} = bulbPow2(z_n) + c + p_mem * z_{n-1}` where `bulbPow2` is the
Mandelbulb square (spherical (r, theta, phi) -> (r^2, 2 theta, 2 phi),
phoenix_core.glsl:62-74). The estimate loop rolls the memory term: it computes
`zNew = bulbPow2(z) + c + pMem * zPrev`, then sets `zPrev = z; z = zNew`
(phoenix_core.glsl:109-114). The DE is the analytic Hubbard-Douady
running-scalar derivative (not a numerical gradient, deliberately, to avoid
ghost-ring artifacts on the non-smooth trig map): the derivative bound is
`dzNew = 2*r*dz + |p_mem|*dzPrev` with `dz_0 = 1, dz_{-1} = 0`
(phoenix_core.glsl:104, 27-30), and `de = 0.5 * r * log(r) / dz`
(phoenix_core.glsl:133). Escape is `|z|^2 > bailout^2` (phoenix_core.glsl:99).
`pMem = 0` collapses to a pure Mandelbulb-Julia; `pMem = -0.5` is the canonical
Phoenix character (phoenix_core.glsl:34-36). Half-cut is the same tilted-plane
CSG idiom (phoenix_core.glsl:137-141).

**Canonical formula / math background:** Ushiki's original 2D Phoenix map is
z_{n+1} = z_n^2 + c + p*z_{n-1} (a second-order / non-Markovian recurrence). The
historically-cited canonical constants are c = 0.5667 (real) and p = -0.5
(sometimes written k = -0.5). Note: secondary sources disagree on the discovery
year (1988 vs 1998); the IEEE Transactions on Circuits and Systems paper is most
commonly dated 1988 (unverified against the primary paper). Community parameter
sets from Paul Bourke and Mitch Richling span c roughly in [-0.4, 0.6] (often
complex) and p (k) roughly in [-0.8, +0.3]; examples: (c=0.5667, p=-0.5)
"classic", (c=0.544992, p=-0.47), (c=0.4, k=-0.25), (c=0.6-0.1i, k=-0.8),
(c=-0.4+0.1i, k=0.2955). Caveat: those values are for the 2D z^2 map; Parsec's
3D Mandelbulb-trig lift will not reproduce them one-to-one, so they are
direction-finding hints rather than exact targets.

- Default c.x = 0.4 vs canonical 0.5667: the per-fractal guide should mention the
  canonical real value c ~ 0.5667 so users can reproduce the textbook Phoenix
  look. The default 0.4 is defensible as a 3D-friendly starting shape (the 2D
  c=0.5667 does not transfer exactly through the Mandelbulb-trig lift), but it is
  not the canonical value and should be labelled as such. Recommend offering a
  "classic Phoenix" preset at (c=0.5667, 0, 0), p=-0.5.
- p range stops at +0.5; Bourke/Richling sets include p up to ~0.3 and the
  occasional imaginary k, so +0.5 is adequate. No change needed.
- The docstring date "Ushiki's original 2D formula" is fine, but the discovery
  year is commonly 1988, not 1998 as the PhoenixState comment implies. This is a
  doc nit, not a schema issue; flag for the guide text.

**For best results:**
- Start from p = -0.5 (canonical). Sweep p slightly toward 0 to watch the curling
  memory growth relax into crystalline Mandelbulb-Julia form.
- Try the canonical c ~ 0.5667 on the real axis (c.y = c.z = 0) for the textbook
  Phoenix flame; add small c.z for 3D asymmetry.
- Keep Iterations modest (14-20) for interactive sweeps; bump for stills.

## Biomorph (`FractalType.Biomorph`)

**What it is:** Pickover biomorphs (Clifford Pickover, 1986) are Julia-type sets
rendered with a componentwise (L-infinity) escape test instead of the usual
rotation-invariant |z| > B. Testing the real and imaginary (here, x/y/z) axes
separately makes orbits "leak out" along individual axes, producing limb-like,
creature-resembling forms with arms, antennae, and bulbous bodies (Pickover
called the radially-symmetric ones "radiolarians"). The effect famously arose
from Pickover swapping AND/OR in the escape condition.

**How Parsec computes it:** Iteration is the plain Mandelbulb square Julia
`z_{n+1} = bulbPow2(z_n) + c` (biomorph_core.glsl:92), but escape uses the
componentwise L-infinity test `max(|z.x|,|z.y|,|z.z|) > bailout`
(biomorph_core.glsl:84-85) - the comment calls this "the entire trick." The DE
is the Hubbard-Douady running-scalar derivative `dz = 2*r*dz` (dz_0 = 1),
`de = 0.5 * r * log(r) / dz` (biomorph_core.glsl:90, 108). The core notes the
standard DE assumes |z|>B escape, so for L-infinity it is approximate (off by at
most a sqrt(3) factor), compensated by the fudge slider
(biomorph_core.glsl:25-29). Default c = (-0.5, 0.5, 0), B = 10
(biomorph_core.glsl:18). Half-cut is the standard tilted-plane CSG
(biomorph_core.glsl:112-116).

**Canonical formula / math background:** Biomorph escape: a point is "interior"
unless |Re(z)| > B OR |Im(z)| > B (componentwise), i.e. the L-infinity norm
exceeds the threshold. Pickover's canonical threshold is B = 10
(madteddy.com confirms "modulus > 10, or magnitude of either the real or
imaginary part > 10"). Typical iteration counts are ~100 (10 is minimal).
Common biomorph constants (for z^n + c radiolarians): z^3 -> c = 0.5; z^5 ->
c = 0.75669(1+i); z^7 -> c = 0.60595(1+i); and exponential variants with c in
0 .. 0.5+0.5i. The specific c = (-0.5, 0.5) used as the Parsec default is a
known biomorph constant in the same family (a clear radial multi-armed creature
per the Parsec docstring; the exact (-0.5, 0.5) pairing is community/empirical
rather than a single canonical citation, so treat as unverified-but-reasonable).
- https://www.madteddy.com/biomorph.htm
- https://www.codeproject.com/Articles/1217543/The-astounding-Pickovers-biomorphs
- https://en.wikipedia.org/wiki/Pickover_stalk

  is iteration-sensitive (the L-infinity leak needs several iterations to express

  guide should recommend raising Iterations to 40-100 for detailed stills. Not a
  range problem (max 500 covers it), only a default/expectation note.
- Bailout B max of 20 is adequate; Pickover canonical 10 is the default. No change.
- c = (-0.5, 0.5) is a reasonable, community-style biomorph constant but I could
  not tie it to a single canonical citation; mark "unverified" in the guide and
  optionally offer radiolarian presets (e.g. c.x ~ 0.5).

**For best results:**
- Keep Bailout at 10 (Pickover canonical); drop toward 3-5 for tighter, denser
  bodies or push to 15-20 for long spindly antennae.
- Raise Iterations to ~50-100 for crisp limbs in final renders.
- Sweep c around (-0.5, 0.5) in small steps; the creature morphology is very
  sensitive, so small c moves give large shape changes.

## Mosely Snowflake (`FractalType.Mosely`)

**What it is:** The Mosely snowflake is a cube-based Sierpinski-Menger family
fractal named after Jeannine Mosely (who discovered the snowflake-sponge variant
around 2006 while exhibiting her business-card Menger sponge). It is built by a
3x3x3 cube subdivision that keeps a subset of subcubes and recurses. Parsec's
core keeps the 8 corner subcubes (the "corner rule"), giving fractal dimension
log8/log3 approximately 1.893; the silhouette down the [1,1,1] body diagonal
reproduces the 3-fold Koch snowflake outline.

**How Parsec computes it:** This is an exact linear cube-IFS, not an escape-time
fold. `estimate()` (mosely_core.glsl:71) iterates per step
(mosely_core.glsl:87-118): (1) world-frame octahedral fold `z = abs(z)`; (2)
rotate into the [111] orthonormal frame U/V/W (mosely_core.glsl:55-57, 92);
(3) twist about the body diagonal via a 2x2 rotation `twistR` on the x'y' plane
(mosely_core.glsl:80, 95); (4) optional kaleidoscope wedge fold of x'y' into a
sector (mosely_core.glsl:98-103, disabled when `wedge >= 2*pi - 1e-4`); (5) back
to world frame; (6) scale toward the (1,1,1) corner
`z = scale*z - (scale-1)` and `dz *= scale` (mosely_core.glsl:109-110). Because
every op except the one uniform scale is an isometry, the DE stays EXACT:
`sdBox(z, vec3(body)) / dz` (mosely_core.glsl:120) - no log-potential
approximation. `scale = 3.0` and `body ~ 1.4` are the snowflake defaults
(mosely_core.glsl:34). Twist near 120 deg breaks mirror symmetry into a chiral
pinwheel; shrinking wedge below 360 deg makes a radial mandala.

**Canonical formula / math background:** The Mosely snowflake is a member of the
Menger-sponge fractal family. Wikipedia describes two cube-removal variants:
a lighter one (remove 8 corners and the centre -> keep 18 -> dim ln18/ln3 ~ 2.631)
and a heavier one (remove only the 8 corners -> keep 19 -> dim ln19/ln3 ~ 2.680).
Parsec instead KEEPS the 8 corners (an inverse / dual corner-cube IFS), which has
the lower dimension log8/log3 ~ 1.893 and the 3-fold Koch silhouette down [111];
this is the construction the Parsec docstring and core describe and is
internally consistent. The standard Menger sponge for reference has dimension
log20/log3 ~ 2.7268. The level-3 business-card Menger sponge (66,048 cards,
finished 2005) is Mosely's famous physical model.

- https://en.wikipedia.org/wiki/Jeannine_Mosely
- https://mathworld.wolfram.com/MengerSponge.html
- https://www.cabinetmagazine.org/issues/24/wertheim_mosely.php

## Cross-fractal notes

- All four cores share the std430 `FoldParams` buffer (binding 1) and the same
  half-cut CSG idiom (`de = max(de, dot(p,n) - planeOffset)`), reinterpreting the
  generic slots per fractal. Schema groups are consistent across the four.
- Defaults are mathematically anchored where a canonical value exists (Apollonian
  Tangency 1.0, Phoenix p -0.5, Biomorph B 10, Mosely Scale 3.0 / Twist 0 /
  Wedge 360). The two soft spots are Phoenix c.x (0.4 vs canonical 0.5667) and
  Biomorph Iterations (16 vs Pickover's typical ~100), both of which are
  defensible performance/3D-lift choices but deserve a note in the guide.
- One genuine code finding: `MoselyState.Reset()` omits `WedgeDeg` and `Fudge`,
  so Reset does not fully restore documented defaults. Flagged, not fixed.
